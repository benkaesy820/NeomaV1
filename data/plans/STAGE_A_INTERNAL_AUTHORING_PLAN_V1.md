# Stage A Internal Authoring Plan v1

## Purpose

Create five million high-information tokens that public source pools cannot target precisely. All material is planning-only until admitted by a later review manifest.

## Rules

- No hidden chain-of-thought or private reasoning traces.
- No copying or paraphrasing evaluation prompts.
- No unsupported factual claims.
- No capability inflation in self-knowledge.
- No secret-shaped values or personal information.
- Every record receives a stable ID, family ID, component, creation method, parent source IDs where applicable, hash, reviewer state, and training permission.
- Generate in small audited batches. Do not create five million tokens in one unreviewed pass.

## Components

### Developer dialogue — 1.5M

Short one- to four-turn developer conversations covering clarification, assumptions, corrections, follow-up constraints, evidence requests, plans, and safe confirmations. Keep most documents within the intended context tier.

### Constraint English — 1M

Semantically verified paraphrase families for negation, missing versus empty, valid falsy values, quantities, boundaries, ordering, sequence, references, conflicts, and preservation requirements. Keep lexical diversity while testing meaning equivalence.

### Bug, review, and commit language — 1M

Pair symptoms with evidence, expected versus actual behavior, severity-calibrated review comments, regression descriptions, and concise commit summaries. Tie code-bearing records to verified snippets.

### CLI, configuration, and errors — 750k

Usage text, option descriptions, configuration comments, validation failures, dry-run summaries, safe warnings, exit-status explanations, and troubleshooting notes.

### Neoma self-knowledge — 500k

Generate from a versioned model card, not free-form invention. Cover model identity, current context, supported languages, data stages, lack of internet or file access unless tools provide it, uncertainty, memory limits, and known weaknesses. Regenerate for each released checkpoint.

### Verified transformations — 250k

Derive short docstrings, summaries, test names, error explanations, and commit descriptions from approved source records. Keep parent lineage and verify semantic consistency.

## Batch protocol

1. Define one capability and acceptance rubric.
2. Generate a small candidate batch.
3. Run schema, safety, and leakage checks.
4. Execute or parse code-bearing records.
5. Deduplicate within and against approved data.
6. Review a representative sample and every flagged record.
7. Admit only accepted records through a versioned manifest.
8. Stop a family when diversity or marginal quality falls.

## Quality metrics

Track exact and near-duplicate rates, average and tail length, vocabulary diversity, template concentration, source-family concentration, syntax/test success, reviewer acceptance, and development-suite change. Token count alone is not progress.
